## [Unreleased]

### Added

- Project bootstrap: repository structure, Apache-2.0 licensing, CLAUDE.md, domain skills, review agents, comprehensive requirements specification.
- Spec-driven development via [GSD](https://github.com/gsd-build/get-shit-done).
- Twenty-three-algorithm catalogue specified in `docs/requirements.md` §7.
- Three-layer public API design (algorithm functions / `Scorer` / `scan` sub-package).
- Patent screen for Metaphone 3 documented as the "Metaphone 3 Precedent" — patent-encumbered algorithms are excluded.
- `fuzzymatch.Validate(a, b string) []Warning` — returns warnings for problematic-but-non-fatal input shapes (empty input, unequal length where the algorithm cares, no tokens after normalise, all non-ASCII dropped, pathologically large input). Companion to the lenient comparison-data contract: algorithms always produce a value; `Validate` reports whether the value is meaningful. Per Phase 8.5 Q4.
- `Warning` and `WarnKind` types accompanying `Validate`. `WarnKind` constants: `WarnEmptyInput`, `WarnUnequalLength`, `WarnNoTokensAfterNormalise`, `WarnAllNonASCIIDropped`, `WarnPathologicallyLargeInput`. `WarnKind.String()` returns CamelCase matching the constant suffix, per the AlgoID.String naming convention. Per Phase 8.5 Q4 + Q6b.
- `ErrInternalInvariantViolated` sentinel — typed panic value for library-internal invariants that should be impossible in correct usage (e.g. `DefaultScorer()` construction failure on a dispatch-table gap that survived option-time validation). Replaces the bare `panic("...this is a bug...")` at `scorer.go:586` with a structured panic carrying the wrapped internal cause. Consumers seeing this sentinel are observing a library bug, not a usage error, and should file an issue at <https://github.com/axonops/fuzzymatch/issues>. Discriminated via `errors.Is(panicValue.(error), ErrInternalInvariantViolated)`. The sentinel MUST NOT be used to wrap caller-supplied parameter errors. Per Phase 8.5 Gap 5 resolution.
- `docs/best-practices.md` — new consumer-facing document covering the Validate-then-Score idiom, Scorer-pinning patterns, concurrency discipline, error-handling conventions, and the three-layer architecture choice. Per Phase 8.5 Plan 17b (VALIDATE-06 surface 5).
- `examples/validate-input-quality/` — new runnable example demonstrating `fuzzymatch.Validate` against 2-3 input pairs with a meta-test pinning the stdout output byte-for-byte. Per Phase 8.5 Plan 17b (VALIDATE-06 surface 6).
- `tests/bdd/features/normalisation.feature` + `tests/bdd/features/determinism.feature` — new BDD feature files covering `Normalise` (NFC / NFD, diacritic stripping, idempotence, ASCII fast path) and the cross-call determinism contract. Per Phase 8.5 Plan 17b (Gap 3 default — scan + suppression features deferred to Phase 9).
- `github.com/axonops/fuzzymatch/scan` sub-package foundation (Phase 9 Plan 09-01) — the Layer 3 turnkey collection-scan public surface: `scan.Item`, `scan.Config`, `scan.Warning`, `scan.Kind` with `KindWithinGroup` / `KindAcrossGroups` constants and `Kind.String()` returning CamelCase, `scan.DefaultConfig(s *fuzzymatch.Scorer) Config` baking the opinionated `CrossGroupThresholdBoost = 0.05` default plus `CompareIdenticalAcrossGroups = false`, and a `scan.Check` stub returning `(nil, ErrNilScorer)` when `cfg.Scorer` is nil. Sentinel errors `scan.ErrNilScorer`, `scan.ErrInvalidItem`, `scan.ErrInvalidConfig` with the four-section godoc template (What / Common causes / Resolution / Example) per Phase 8.5 Q4. Full `Check` body (within-group + cross-group + bucket + suppression + sort) lands across plans 09-02 through 09-06. Sub-package adds zero non-stdlib runtime dependencies to the root module. Reviewer sign-offs recorded in plan 09-01's SUMMARY.md.
- `fuzzymatch.Scorer.NormalisationOptions() (NormalisationOptions, bool)` — new public accessor returning the Scorer's stored `NormalisationOptions` and a bool indicating whether the Scorer applies normalisation. Mirrors the existing `Threshold()` and `Algorithms()` accessors. Used by the Phase 9 scan sub-package to canonicalise `SuppressedPairs` entries and build token buckets using the same normalisation pipeline the Scorer uses for scoring. By-value return; safe for concurrent use; the Scorer is immutable after `NewScorer`. Resolves 09-RESEARCH.md Open Question 1 (how scan accesses the Scorer's normalisation state without coupling Config to a duplicate `NormalisationOptions` field). Spec amendment to `docs/requirements.md` §8 in lockstep.
- `scan.Check` full body (Phase 9 Plans 09-02 through 09-06) — the Layer 3 collection-scan implementation. Within-group + opt-in cross-group passes (`Config.CompareAcrossGroups`); validation gate (P0..P4) using `errors.Join` to collect-then-report every offending index in a single round-trip (D-03 empty Name + D-04 invalid Config + D-05 empty SuppressedPairs entry + D-06 duplicate `(Name, Group)`); token-bucket pre-filter optimisation (`bucketThreshold = 50` empirically validated on `darwin/arm64`) running in parallel with the naive O(N²) reference and proven equivalent via `PropCheck_BucketEquivalentToNaive` (SCAN-02); three-rule suppression composition via the `isSuppressed` predicate (Rule 1 per-item `SilenceLint`, Rule 2 `SuppressedPairs` canonical-pair lookup, Rule 3 cross-group identical-Name default — SCAN-03 + SCAN-04 merged into a unified pre-emission gate); raw-byte lex canonicalisation of `(NameA, NameB)` with `(GroupA, GroupB, TagA, TagB)` swapping in lockstep, then deterministic `sort.SliceStable` on the 5-tuple `(Kind, NameA, NameB, GroupA, GroupB)`; in-line completeness assertion panicking with `fuzzymatch.ErrInternalInvariantViolated` if duplicate sort keys ever appear (SCAN-05 defence-in-depth — unreachable on D-06-validated input). PERF-05 budget (< 2 s / 10 k items within-only) verified by `BenchmarkScanCheck_DefaultScorer_10k`. Per Phase 9 SCAN-01..06 + PERF-05 + DET-04.

Iteration-order-independent JSON serialisation: `Warning.Scores` is serialised as a sorted-key array of `[AlgoID.String(), float64]` tuples so map iteration randomness never surfaces. `make verify-determinism` runs `TestGolden_ScanDefault` on every platform in the CI matrix.

Stdout golden meta-test (`main_test.go`) mirroring `examples/scorer-composition` shape pins output byte-for-byte across runs and platforms. Per DX-02.
- `scan/example_test.go` runnable godoc Examples (Plan 09-08) — five `// Output:`-gated Example functions covering every demonstrable public scan symbol: `ExampleKind_String`, `ExampleDefaultConfig`, `ExampleCheck` (within-group), `ExampleCheck_acrossGroups` (cross-group with Rule 3), `ExampleCheck_suppression` (all three suppression rules). ### Changed

- `WithThreshold` now rejects NaN (returns `ErrInvalidThreshold`). `±Inf` was already rejected by the existing range check; this change closes the NaN escape. Guard form: `if math.IsNaN(t) || t < 0.0 || t > 1.0 { return ErrInvalidThreshold }`. Per Phase 8.5 Q2.
- `WithAlgorithm` now rejects weight values that are NaN, `±Inf`, or ≤ 0 (returns `ErrInvalidWeight`). Per Phase 8.5 Q2 extension covering all Scorer-construction parameter validation.
- `WithTverskyAlgorithm` now rejects `α + β ≤ 0` at construction time (returns `ErrInvalidTverskyParam`). Previously `WithTverskyAlgorithm(_, 0, 0, _)` constructed successfully then panicked at first `Score` call. The same guard applies to the direct `TverskyScore` call path, which panics with the same sentinel per the data-vs-parameter framework documented in §6.A. Per Phase 8.5 Q2.
- Every exported error sentinel now carries the four-section godoc block (What / Common causes / Resolution / Example) per `.claude/skills/documentation-standards/SKILL.md` § Error sentinel documentation. Per Phase 8.5 Q4.
- `AlgoID.String()` is documented as CamelCase matching the constant suffix (`"Levenshtein"`, `"JaroWinkler"`, `"NYSIIS"`, `"DamerauLevenshteinOSA"`, etc.) — locking the convention against earlier draft language that referenced snake_case forms. Per Phase 8.5 Q6b.
- Monge-Elkan direct calls (`MongeElkanScore`, `MongeElkanScoreAsymmetric`) now panic with `ErrInvalidInnerAlgo` when passed an invalid inner AlgoID (unknown AlgoID, `AlgoMongeElkan` self-reference, or a token-tier AlgoID). This aligns Monge-Elkan with the data-vs-parameter framework locked in Phase 8.5 Q2. Scorer-construction callers receive the same sentinel as a typed error. New sentinel `ErrInvalidInnerAlgo` added to the canonical `errors.go` set. Per Phase 8.5 Q4 follow-up.
- Renamed `ErrInvalidAlgorithm` → `ErrInvalidAlgoID` to match the canonical sentinel name in `docs/requirements.md §6` and resolve the dangling `monge_elkan.go` / `llms-full.txt` non-existent-sentinel Critical finding from the Phase 8 review. Code and spec are now aligned. Per Phase 8.5 Gap 4 resolution.
- `docs/requirements.md` §12.1 + §8 + §19 + §20 amended in lockstep with the Phase 9 plan 09-01 scan sub-package foundation (SPEC OVERRIDE notes added inline, api-ergonomics-reviewer sign-off recorded in plan 09-01's PR per the Phase 7 / Phase 8 precedent):
  - §12.1 Kind type renamed from the originally-specified two-word `WarningKind` name per 09-CONTEXT.md §1 D-02 — `scan.KindWithinGroup` / `scan.KindAcrossGroups` are unambiguous at the call site and avoid accidental symmetry with the root package's `WarnKind` type.
  - §12.1 `Warning.Scores` field type changed from `map[string]float64` to `map[AlgoID]float64` per 09-CONTEXT.md §1 D-01. Extends the Phase 8 ScoreAll override at §8.3 + §8.6 for the same typed-enum-keys rationale: the rest of the library exposes AlgoID and gives consumers compile-time key safety.
  - §12.1 `CrossGroupThresholdBoost` "Default: 0.05" statement migrated from the struct field godoc to `scan.DefaultConfig` godoc per 09-CONTEXT.md §2 D-04, mirroring Phase 8's `DefaultScorer` / `DefaultScorerOptions` pattern. Zero-value of `Config.CrossGroupThresholdBoost` is `0.0`; the opinionated default is supplied by `DefaultConfig`.
  - §12.1 `Check` godoc updated to document the `errors.Join` collect-all multi-error semantics for `ErrInvalidItem` (D-03 + D-06) and `ErrInvalidConfig` (D-05). The original "wrapped with the offending item's index" wording (singular) was inaccurate now that validation collects every offending index in a single round-trip.
  - §8 new entry for `Scorer.NormalisationOptions()` accessor matching the spec convention used for `Threshold()` and `Algorithms()`.
  - §19 + §20 acceptance-criteria rows updated to reflect the Kind rename.
- `docs/requirements.md` §12.1 `Warning.NameA` canonicalisation wording amended (Phase 9 Plan 09-06 close-out): the originally-specified "lexicographically smaller (after normalisation)" tiebreaker is no longer needed because D-06 validates `(Name, Group)` uniqueness at Check entry, guaranteeing the `(Kind, NameA, NameB, GroupA, GroupB)` sort key is a strict total order on every valid input. The implementation uses raw-byte lex on the RAW item Names so consumers see their input Names in the output verbatim. SPEC OVERRIDE annotation added inline.

### Breaking (pre-v1.0)

- `MongeElkanScore` is now **symmetric** by default. The v0.x directional behaviour is available as the new `MongeElkanScoreAsymmetric`. The `NormalisationOptions` parameter has been removed from both functions — it had no effect inside Monge-Elkan. Per Phase 8.5 Q3.
- `PartialRatioScoreRunes` has been removed. Token-tier algorithms operate on the output of `Tokenise`, which is itself rune-aware; the byte-level Indel kernel produces correct results on Unicode post-`Tokenise`. Per Phase 8.5 Q5.
- Three unused error sentinels removed: `ErrInvalidConfiguration`, `ErrInvalidInput`, `ErrEmptyInput`. None had call sites in the library. Per Phase 8.5 Q4.
- `HammingDistance` and `HammingDistanceRunes` now return `int` (no error tuple). On unequal-length input the silent-max policy applies — they return `max(len(a), len(b))`. `HammingScore` continues to return `0.0` on unequal length. This matches existing code behaviour; the spec catch-up removed the placeholder `ErrUnequalLength` sentinel. Per Phase 8.5 Q1.

### Notes

This project is pre-release. The API is not stable until `v1.0.0`. Phase 8.5 (Review Remediation Gate) is the breaking-change consolidation phase ahead of v1.0; further breaking changes are expected to be minimal.
