#### Reusable existing patterns

- **Explicit-parens-for-determinism precedent:** every algorithm that does floating-point reduction (cosine, scorer, jaro at jaro.go, lcsstr.go) already uses explicit `(x*y) + z` parens with rationale comments. The Q11b pattern extends this with a second outer `float64(...)` cast that the Go compiler treats as a forced rounding point and therefore cannot fuse into FMA.
- **Reference:** `golang/go#17895` "go vet: detect fused multiply-add inadvertently introduced by the optimiser" — the canonical Go-team-acknowledged context for FMA-defeat patterns. The double-cast technique is the established workaround per the issue's discussion.

#### Pitfalls specific to fuzzymatch

- **The change is COSMETICALLY SMALL but determinism-CRITICAL.** Two file edits, ~4 line changes total, ~30 lines of comment updates. Plan the patch + the comment rewrite as one atomic commit.
- **Cross-platform CI verification REQUIRED.** The Phase 4 / 5 / 8 cross-platform matrix has been green for cosine.go without the outer cast — meaning either (a) the existing `qa[k]/qb[k]` integer-derived values are small enough that FMA-vs-non-FMA divergence falls below the byte-diff threshold (as the existing comment hedges), or (b) the compiler hasn't been emitting FMA on those specific input shapes. Q11b is a **belt-and-braces** defence ahead of v1.0 stability rather than a known-failing fix. The verification gate is the cross-platform matrix re-running TestGolden_Cosine after the change — if any divergence appears across linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64, that's a regression. Recommended to commit Q11b on a branch and run the matrix manually before merging.
- **`bench.txt` consistency:** if Q11b changes the cosine reduction order subtly, micro-benchmarks may move by 1–2 %. Whether the change crosses the 10 % benchstat threshold is uncertain — the planner should sequence Q11b BEFORE Q13's bench.txt regeneration step.

#### Reusable existing patterns

- **Stack-allocated buffer pattern:** `soundex.go` and `normalise.go` already use `var stackBuf [N]byte` for ASCII fast paths with the `string(stackBuf[:n])` finaliser. Q7a's DoubleMetaphone `[4]byte` accumulator follows the same idiom — substitute `strings.Builder` for `[dmMaxLen]byte` + an `int` length counter; replace `b.WriteByte(c)` with `buf[n] = c; n++`; replace `b.String()` with `string(buf[:n])`.
- **ASCII fast path pattern:** `normalise.go` has the canonical ASCII fast path — detect via `unicode.IsASCII` (or a faster all-bytes-≤127 byte loop), emit bytes; non-ASCII falls back to `[]rune` path. Q8b Tokenise extends this idiom.
- **Zero-copy substring pattern:** Go-idiomatic; `s[lo:hi]` is a header-only operation that shares the input string's backing array. When the call site never mutates the input, this is safe and zero-alloc. Q8b applies this to the no-lowercase case.

#### Pitfalls specific to fuzzymatch

- **Q8b is the trickiest item in cluster 5.** The current `Tokenise` does:
  1. `runes := []rune(s)` — one allocation
  2. Walks `runes`, detecting separator/camelCase boundaries
  3. Calls `appendToken(tokens, runes[start:i], opts.Lowercase)` — which itself does `string(rune slice)` allocating one per token, plus lowercase folding if requested
  
  The Q8b lock says: when input is ASCII AND `opts.Lowercase == false`, boundaries become byte-index ranges and tokens are returned as `s[lo:hi]` substrings (zero-copy). When `opts.Lowercase == true`, a single scratch `[]byte` buffer of length `len(s)` is allocated once, ASCII-lowercased in-place, then `s[lo:hi]` returns substrings of THAT buffer (one allocation for the buffer; the substrings are zero-copy on top of it).
  
  Implementation risks:
  - **Backing-array escape:** when `s[lo:hi]` is returned as part of `[]string`, the entire `s` (or scratch buffer) backing array is kept alive. For huge inputs, this is a memory-retention foot-gun. The existing `extractQGrams` comment (per Phase 8 review Important findings) flags this for q-grams. Tokenise must document the same: returned tokens retain a reference to the input string until all tokens are released.
  - **camelCase boundary detection on bytes:** uppercase/lowercase ASCII letters detected by `(c >= 'A' && c <= 'Z')` / `(c >= 'a' && c <= 'z')` — straightforward. But ASCII-fast-path-only fires when input contains no multibyte UTF-8 (verified by `unicode.IsASCII` or a byte-loop check). The non-ASCII path stays at the rune-based implementation.
  - **Backward-compatible boundary semantics:** the ASCII fast path MUST produce IDENTICAL token sequences to the rune path on ASCII input. Property test required: `PropTokenise_ASCIIFastPathEquivalent` asserting `Tokenise(s, opts) == TokeniseRunePath(s, opts)` for any ASCII `s`.
  - **Lowercase folding side-effect:** when `opts.Lowercase == true`, the scratch buffer is the lowercased copy. Future tokens returned share THIS buffer. If a caller mutates `opts.Lowercase` between calls — no, Tokenise doesn't accept mutable options. Safe.
  
- **Q8b benchmark + BDD requirement:** the CONTEXT.md says "Dedicated plan with BDD + benchmarks." This means at least: (a) `BenchmarkTokenise_ASCII_FastPath_Short/Medium/Long` to verify the alloc-count drop, (b) a BDD scenario in `tokenise.feature` or `normalisation.feature` asserting ASCII fast-path output equality with the rune path.

- **Q7a DoubleMetaphone re-benchmarking:** after the `[4]byte` change, `BenchmarkDoubleMetaphoneKeys_*` allocs/op drops from ~6 (currently) to ~2. `BenchmarkDefaultScorer_*` (which composes DoubleMetaphone) drops correspondingly. **This is the dependency for Q8c.** Order: Q7a → Q8c verification.
