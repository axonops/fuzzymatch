<objective>
Q8b — implement the Tokenise ASCII fast path. Per CONTEXT.md, this is a dedicated plan because it requires its own BDD scenarios + benchmarks and the property test for ASCII-vs-rune-path equivalence is load-bearing for token-tier algorithm correctness.

The current Tokenise:
1. `runes := []rune(s)` — one allocation
2. Walks `runes`, detecting separator/camelCase boundaries
3. Calls `appendToken(tokens, runes[start:i], opts.Lowercase)` — per-token alloc, plus lowercase folding

The Q8b target:
- When input is ASCII AND `opts.Lowercase == false`: boundaries become byte-index ranges; tokens returned as `s[lo:hi]` substrings (zero-copy on top of input).
- When input is ASCII AND `opts.Lowercase == true`: one scratch `[]byte` buffer of length `len(s)` allocated once, ASCII-lowercased in-place, then `scratch[lo:hi]` returns substrings of THAT buffer (one allocation total).
- Non-ASCII falls back to the existing rune-based implementation.

Token-tier algorithms (TokenSort, TokenSet, PartialRatio, TokenJaccard, MongeElkan) inherit the savings.

**Implementation risk:** backing-array escape. Tokens retain a reference to input `s` (or to the scratch buffer) until all tokens are released. For huge inputs, this is a memory-retention concern. The godoc MUST document this scope explicitly.

Output: new ASCII fast path in `tokenise.go`, an equivalence property test, dedicated benchmarks asserting the alloc-count drop, and a BDD scenario asserting ASCII-vs-rune-path equality.
</objective>

1. Refactor `Tokenise` to detect ASCII input early:
   - At function entry, scan `s` for any byte > 127. The fastest pattern is a simple byte-loop: `for i := 0; i < len(s); i++ { if s[i] > 127 { ... fall back to rune path ... break } }`. (Or: use `unicode.IsASCII` per code point with `range s` — but the byte-loop is faster for the strict-ASCII detection case.)
   - If non-ASCII detected, jump to the existing rune-based implementation (preserve current behaviour).
2. ASCII fast path (when input is all-ASCII):
   - **Case A: `opts.Lowercase == false`**:
     - Iterate `s` byte-by-byte detecting separators (`-`, `_`, `.`, ` `, etc. — match the existing rune-path separator set) AND camelCase boundaries (`(c >= 'A' && c <= 'Z')` after `(c >= 'a' && c <= 'z')`).
     - Emit tokens as `s[lo:hi]` substrings — zero-copy.
   - **Case B: `opts.Lowercase == true`**:
     - Allocate ONE scratch buffer: `scratch := make([]byte, len(s))`.
     - Walk `s` once, copying bytes into `scratch` with ASCII lowercase folding: `if c >= 'A' && c <= 'Z' { scratch[i] = c + 32 } else { scratch[i] = c }`.
     - Then run the same boundary-detection loop on `scratch`, emitting tokens as `scratch[lo:hi]` (substrings of the scratch buffer — zero-copy on top of the one allocation).
     - Convert to `[]string` via the stdlib path: `tokens = append(tokens, string(scratch[lo:hi]))`. Go's `string([]byte)` always allocates and copies. **Constraint:** the locked "single scratch buffer" semantic refers ONLY to the single lowercase-pass scratch buffer; per-token string conversion alloc is structurally unavoidable without `unsafe.String`.
     - **LOCKED DECISION (m11 revision 2026-05-17):** DO NOT use `unsafe.String`. `.claude/skills/go-coding-standards/SKILL.md` is silent on the `unsafe` package; the correctness-first project default is AVOID. CLAUDE.md "Architecture" explicitly forbids cgo and the spec prohibits any non-stdlib runtime dep; the spirit of those constraints extends to `unsafe`. The achieved alloc budget is locked at: **1 alloc per Tokenise call** for the lowercase-pass scratch buffer (independent of token count), plus 1 alloc per emitted token for the `string(scratch[lo:hi])` conversion. This is documented as the v1.0 budget; tightening to truly-zero-copy via `unsafe.String` is deferred to a post-v1.0 issue if profiling justifies it.
3. **Godoc scope note** at the public Tokenise function: add a paragraph explaining the memory-retention scope:
   "Performance note (Q8b): on ASCII input, returned tokens are substrings of either the input string (when opts.Lowercase == false) or a single internal scratch buffer (when opts.Lowercase == true). The returned []string retains a reference to that backing array until all token strings are released; for very large inputs this is a memory-retention consideration. Non-ASCII input falls back to the rune-based path, which allocates per token."
4. Add British English throughout new godoc.
