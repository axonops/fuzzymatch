## Accomplishments

- **Q7a:** DoubleMetaphoneKeys now uses two stack-allocated `[dmMaxLen]byte` buffers + `int` length counters instead of `strings.Builder` pairs. dmAdd helper refactored to pointer-to-counter parameters. Byte-count drops from 24 B/op → 16 B/op on the canonical "Schmidt" benchmark; wall time drops from ~119 ns/op → ~98 ns/op (~18%).
- **Q7d:** extractQGrams + extractQGramsRunes pre-size their result maps with 25% capacity headroom (`cap = windowCount + windowCount/4`). Reduces map-rehash probability on medium-to-long inputs per RESEARCH.md §4.2.
- **Q11e:** TestDamerauLevenshteinFullScore_ZeroAllocs_ASCII_Short renamed and un-skipped — now `TestDamerauLevenshteinFullScore_ShortAllocBudget_ASCII`, asserting ≤ 1 alloc/op via `testing.AllocsPerRun(100, ...)`. Locks the Q8a budget at test time.
- **Q7c:** Long-input heap-fallback scope notes added to public-function godoc on DamerauLevenshteinFullDistance, RatcliffObershelpScore, and LongestCommonSubstring — each cross-references docs/requirements.md §14.1 for the long-input row.
- **Q7b:** ≤ 1 alloc floor (not 0) scope notes added to SoundexCode and Normalise godoc, documenting why the 0-alloc target was unachievable (unsafe.String excluded by project policy).
- **Q7b/Q8a/Q8c/Q8d:** Benchmark file godoc updated to match revised §14.1/§14.2/§14.3 budgets in 5 bench files (damerau_full, ratcliff_obershelp, scorer, soundex, normalise).

## Q7a Alloc Profile — Before / After

Recorded for SUMMARY traceability per plan output requirement.

| Benchmark | Before | After | Delta |
|---|---|---|---|
| BenchmarkDoubleMetaphoneKeys_ASCII_Short | 3 allocs, 24 B/op, 118.5 ns/op | 3 allocs, 16 B/op, 97.87 ns/op | bytes −33%, time −17% |
| BenchmarkDoubleMetaphoneKeys_ASCII_Medium | 3 allocs, 32 B/op, 125.6 ns/op | 3 allocs, 21 B/op, 107.8 ns/op | bytes −34%, time −14% |
| BenchmarkDoubleMetaphoneKeys_ASCII_Long | 4 allocs, 112 B/op, 174.2 ns/op | 4 allocs, 104 B/op, 157.0 ns/op | bytes −7%, time −10% |
| BenchmarkDoubleMetaphoneScore_ASCII_Short | 6 allocs, 48 B/op, 234.3 ns/op | 6 allocs, 32 B/op, 199.9 ns/op | bytes −33%, time −15% |
| BenchmarkDoubleMetaphoneScore_Identity (short-circuit) | 0 allocs, 0 B/op, 2.41 ns/op | 0 allocs, 0 B/op, 2.42 ns/op | unchanged |

Note: alloc count is unchanged because `strings.Builder.String()` does NOT copy (it returns the buffer via unsafe internally) — the prior assumption that Builder allocated 2× per call was incorrect. Each DoubleMetaphoneKeys call structurally needs 3 allocs: dmPrep result + primary key string + secondary key string. The win is in byte-count + wall time, not alloc count.
