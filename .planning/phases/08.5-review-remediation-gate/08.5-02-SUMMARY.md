# Phase 08.5 Plan 02: MongeElkan symmetric-by-default rename + opts removal Summary

Q3 — atomic rename of the Monge-Elkan public surface across 20 .go files plus the two `llms*.txt` AI-friendly artefacts. The unsuffixed `MongeElkanScore` now denotes the symmetric default; the v0.x asymmetric direct surface is now `MongeElkanScoreAsymmetric`; the previously-inert `NormalisationOptions` parameter has been removed from both surfaces. Breaking pre-v1.0; CHANGELOG already records the surface flip.

## Deliverables

- `monge_elkan.go` — symmetric default at the unsuffixed name; directional surface renamed; opts parameter dropped; godoc rewritten on both functions including the file-level synopsis, the algorithm-derivation block, the public-surface block, the asymmetry-discriminating reference-vector block (RV-ME6 keystone), and the NormalisationOptions-removal note.
- `dispatch_monge_elkan.go` — dispatch binding now calls `MongeElkanScore(a, b, AlgoJaroWinkler)`; file-level godoc reframed to reference the symmetric default; opts forwarding removed.
- `monge_elkan_test.go` — `TestMongeElkanScoreAsymmetric` (was `TestMongeElkanScore`) covers RV-ME1..RV-ME6 derivations on the directional surface; `TestMongeElkanScoreAsymmetric_DirectionSensitive` (was `_AsymmetryDirectionSensitive`) is the load-bearing direction-swap regression gate; `TestMongeElkanScore` (was `TestMongeElkanScoreSymmetric`) pins symmetric-default identity, both-empty, one-empty, construction-from-directional-mean, and order-independence; `TestMongeElkan_PanicsOnNonPermittedInner` exercises both surfaces; `TestMongeElkan_PanicMessageFormat` uses the directional surface; `TestMongeElkanScore_DispatchRegistration` verifies the dispatch wrapper matches the symmetric default; `TestMongeElkanScoreAsymmetric_BinaryInner_{Soundex,DoubleMetaphone,NYSIIS,MRA}` exercise binary-inner composition on the directional surface.

## Key Decisions

1. **Atomic single-commit rename** — per the plan's `<action>` discipline, the rename + opts removal lands in one `feat!:` commit. The grep-based acceptance criteria (`MongeElkanScoreSymmetric` count = 0; `opts NormalisationOptions` in MongeElkan signatures count = 0) pass against the post-commit working tree.
2. **Plan scope extended via Rule 3** — the plan's `files_modified` listed 9 files; compile-green required also editing `scorer_options.go`, `errors.go`, `tests/bdd/steps/algorithms_steps.go`, `examples/identifier-similarity/main.go`, plus the property/golden/cross-validation test files (`props_test.go`, `algorithms_golden_test.go`, `monge_elkan_cross_validation_test.go`, `scorer_options_test.go`), plus the dispatch comment-only references in `dispatch_mra.go` and `dispatch_nysiis.go`, plus the `llms.txt` / `llms-full.txt` AI-friendly artefacts (the `TestAIFriendly_LLMSTxtReferencesEveryExportedSymbol` test enforces sync). All 11 extra files updated in the same atomic commit.
3. **Gherkin language preserved** — the BDD feature file `tests/bdd/features/monge_elkan.feature` says `"MongeElkan score"` (semantically: the directional surface — that's what direction-sensitivity scenarios test) and `"MongeElkanSymmetric score"` (the symmetric default — order-independence scenarios). The Go-side step methods were rewired to point at the renamed Go API while the Gherkin step text is unchanged. Rationale: avoid touching the BDD `.feature` files unnecessarily; the Go-side mapping is the only thing that needs to change for the rename to land cleanly.
4. **Example Output lines numerically unchanged** — the rename is a SWAP of names, not of semantics. The body of `ExampleMongeElkanScore` (old, asymmetric) keeps its numerical output when renamed to `ExampleMongeElkanScoreAsymmetric` because the function being called still computes the same thing (just renamed). Same for the symmetric variant. So the Output lines were verified rather than recomputed.

## Semantic-Flip Audit Notes (cross_algorithm_consistency_test.go)

The plan flagged `cross_algorithm_consistency_test.go` lines 737-766 as the highest-risk semantic-flip surface. Audit results:

- **Line 737 (was `asymID := MongeElkanScore(idA, idA, ...)`):** now `asymID := MongeElkanScoreAsymmetric(idA, idA, AlgoJaroWinkler)`. Identity short-circuit preserved on the directional surface — bit-exact 1.0.
- **Line 738 (was `symID := MongeElkanScoreSymmetric(idA, idA, ...)`):** now `symID := MongeElkanScore(idA, idA, AlgoJaroWinkler)`. Symmetric-default identity — bit-exact 1.0 because both directions yield 1.0 and the mean of two 1.0s is 1.0.
- **Lines 752-755 (the construction pin):** `fwd := MongeElkanScoreAsymmetric(asymInputA, asymInputB, AlgoJaroWinkler)` + `rev := MongeElkanScoreAsymmetric(asymInputB, asymInputA, AlgoJaroWinkler)` + `sym := MongeElkanScore(asymInputA, asymInputB, AlgoJaroWinkler)`; expected `(fwd+rev)/2 == sym` to tolerance 1e-12.
- **Line 765 (direction-divergence gate):** `if fwd == rev` test now reads "the directional surface MongeElkanScoreAsymmetric is INTENTIONALLY direction-sensitive". Semantically identical to the pre-rename assertion — the asymmetric surface should not collapse to symmetric behaviour.

Net audit conclusion: every assertion is unchanged in MEANING; only the function names change. No test-expectation values needed regenerating.
