## Accomplishments

- **Task 1 — tokenise.go ASCII fast path:** Tokenise() now dispatches at entry on `isASCIITokenise(s) && !containsNonASCII(opts.SeparatorChars)`. The fast path operates byte-by-byte:
  - `Lowercase=false`: tokens are emitted as `s[lo:hi]` substrings — zero allocation per token; the only alloc is the `[]string` header (and slice-growth slack when the input has many tokens).
  - `Lowercase=true`: a single `make([]byte, len(s))` scratch buffer is filled once with `s` folded to lowercase via the bitwise OR 0x20 ASCII rule; tokens are then emitted as `string(scratch[lo:hi])` — one allocation per emitted token for the canonical string conversion (structurally unavoidable without `unsafe.String`, excluded by policy).
  - Non-ASCII input or non-ASCII `SeparatorChars` falls back to the preserved `tokeniseRune()` (the pre-Q8b rune implementation, now a private function).
  - Boundary-detection mirrors the rune path byte-for-byte: `isCamelBoundaryASCII` and `isUpperRunTrailingASCII` are the byte-level analogues of the rune-level helpers, using the existing `isUpperASCII` / `isLowerASCII` helpers from `normalise.go`.
- **Task 1 — Memory-retention scope godoc:** Tokenise's public godoc gains a "Performance note (Phase 8.5 Q8b)" paragraph explaining that on ASCII input the returned tokens retain a reference to either `s` or the scratch buffer until released. Consumers who need the input string to be GC-eligible can `strings.Clone(token)` (Go 1.18+). Threat T-08.5-16 mitigation.
- **Task 2 — TestProp_Tokenise_ASCIIFastPathEquivalent:** 500-seed × 8 option-bitfield (Lowercase × SplitCamelCase × SplitConsecutiveUpper) property test using a custom `asciiTokeniseInput` generator biased to letters/digits/separators. Asserts `Tokenise(s, opts) == TokeniseRuneForTest(s, opts)` byte-identical via `reflect.DeepEqual`. Load-bearing correctness gate per threat T-08.5-17.
- **Task 2 — Benchmarks:** Five new benchmarks land in `tokenise_bench_test.go`:
  - `BenchmarkTokenise_ASCII_FastPath_Short` (21-byte camelCase identifier)
  - `BenchmarkTokenise_ASCII_FastPath_Medium` (~80-byte mixed style)
  - `BenchmarkTokenise_ASCII_FastPath_Long` (~530-byte compound identifier, Lowercase=false)
  - `BenchmarkTokenise_ASCII_FastPath_LowercaseLong` (same input, Lowercase=true — documents the upper-bound v1.0 budget excluding unsafe.String)
  - `BenchmarkTokenise_Unicode_FastPath_Long` (1 non-ASCII byte at front forces rune path; documents the alloc-count delta between paths)
- **Task 2 — BDD scenario:** New `@tokenise @ascii_fast_path` scenario in `scorer.feature` asserts the fast and rune paths produce `["foo", "Bar", "baz", "qux"]` on `"fooBar-baz_qux"` under `SplitCamelCase=true / Lowercase=false`. The rune-path invocation is forced by augmenting `SeparatorChars` with `"™"` — the documented dispatch criterion in `Tokenise()`. Five step definitions and a small slice-literal parser land in `scorer_steps.go`.
- **Task 2 — export_test.go:** `TokeniseRuneForTest = tokeniseRune` exposes the rune path to the external `fuzzymatch_test` package so the property test can compare paths without polluting the public API.

**Lowercase=true upper bound:** ~140 per-token string conversions remain (no `unsafe.String` per policy); the absolute alloc-count drop (158 → 147) reflects only the single scratch buffer collapsing the per-token lowercase-fold allocations. Documented in tokenise.go Performance note godoc; tightening deferred to post-v1.0 issue if profiling justifies it.

## Decisions Made

- **Internal entry point preserved:** `tokeniseRune` remains private; the property test reaches it via the `_ForTest` export pattern (established in `export_test.go`). Avoids polluting the public API while satisfying the equivalence-test oracle requirement.
- **Dispatch gate is two-pronged:** Both `isASCIITokenise(s)` AND `!containsNonASCII(opts.SeparatorChars)` must hold for the fast path to engage. A non-ASCII separator string would force the rune path even on an ASCII input. Documented in `Tokenise()` godoc and exploited by the BDD step to force the rune path on ASCII input.
- **No unsafe.String:** Per locked m11 revision (2026-05-17), the Lowercase=true path retains a structural per-token alloc floor (the canonical `string([]byte)` conversion). The achieved v1.0 alloc budget is documented in the public godoc; a post-v1.0 issue can revisit unsafe.String if profiling justifies it.
- **Boundary decisions on original-case bytes:** Even on the Lowercase=true path, the boundary-detection scan operates on `s` (the original case), not on the lowercased scratch buffer. This mirrors the rune path's "lowercase after boundary detection" semantic — boundaries between A..Z and a..z must be visible to the detector. Tokens are then emitted from the scratch buffer at the boundary indices (the lowercase fold is byte-position-preserving so indices are valid in both buffers).
- **BDD rune-path forcing strategy:** Appended `"™"` (U+2122, 3-byte UTF-8) to `SeparatorChars`. Never appears in plausible inputs; only effect is to flip the `containsNonASCII()` check. Documented inline in the Gherkin scenario block.
