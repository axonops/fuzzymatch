## 5. Improvement-Tier Residue

Per Q14b Path A residue handling, the following Improvement-tier items are documented with reasoned acceptance. None block Phase 9.

### 5.1 R1 — Q7d 25 % q-gram capacity-hint regression (algorithm-performance)

**Finding:** Plan 16 SUMMARY notes Q7d's 25 % capacity hint on q-gram maps surfaced as a **+15-19 % time / +90-107 % bytes regression on Medium/Long ASCII** benchmarks.

**Reasoned acceptance:** The capacity hint was added to reduce **rehash probability** at Medium, which is correctness-adjacent (rehash mid-iteration is undefined behaviour territory for very rare patterns). The Medium/Long regression on ASCII is the trade-off the lock accepted. Closure path: a future v1.x performance pass can revisit the heuristic (potentially a non-ASCII-only capacity hint, or a different threshold curve).

**Scheduled:** v1.x post-1.0.

### 5.2 R2 — Gap 3 `scan.feature` + `suppression.feature` deferred to Phase 9 (bdd-scenario)

**Finding:** 2 of 4 originally-missing BDD feature files are absent.

**Reasoned acceptance:** Both feature files describe **Phase 9 (Collection Scan sub-package)** surface area. The `scan` sub-package itself does not exist yet — Phase 9 builds it. Writing feature files for code that doesn't exist would be premature. The deferral was the explicit recommended default per Phase 8.5 CONTEXT.md §Gap 3:

> Recommended split: `normalisation.feature` and `determinism.feature` land in Phase 8.5; `scan.feature` and `suppression.feature` are deferred to Phase 9 with an explicit acknowledgement in the verification-gate report ("two Critical BDD findings deferred to Phase 9 because the feature files describe Phase 9 surface").

**Scheduled:** Phase 9 plan kickoff.

### 5.3 R3 — Coverage floor residue (test-analyst)

**Finding:** Three coverage gaps surface when running `make check`:

| Floor | Threshold | Actual | Offenders |
|---|---|---|---|
| **Floor 1** — overall | ≥ 95.0 % | 92.1 % | (aggregate) |
| **Floor 2** — per-file | ≥ 90.0 % | 70.1 % / 81.8 % | `double_metaphone.go` (70.12 %), `nysiis.go` (81.82 %) |
| **Floor 3** — per-exported-function | ≥ 90.0 % | (6 violations) | `DamerauLevenshteinOSADistanceRunes` (80.0 %), `DefaultScorer` (75.0 %), `DoubleMetaphoneKeys` (65.7 %), `NYSIISCode` (80.5 %), `NewSWGParams` (75.0 %); `ScorerAlgorithm` (exported type not referenced in any `*_test.go`) |

**Reasoned acceptance:**

- **None of these gaps are net new regressions introduced by Phase 8.5.** They are inherited from the Phase 8 baseline. Plan 11's SUMMARY explicitly listed them as "6 Floor-3 violations (5 funcs + 1 type) — input to Plan 13's improvement sweep." Plan 13 did not close them.
- **The phonetic encoders are inherently branchy.** `DoubleMetaphone.DoubleMetaphoneKeys` has cyclomatic complexity 275 (per `golangci_residue.md`) because it implements the canonical Philips 2000 state machine — every branch is rule-anchored in the primary source. Hitting 90 % statement coverage requires either (a) generating exhaustive synthetic test names that exercise every rule branch, or (b) extracting reference vectors from the SWI-Prolog C reference more exhaustively. Both are tractable but were out of scope for Phase 8.5.
- **The library overall coverage is 92.1 %** — strong against the industry baseline. The 95 % aspirational floor is per `.claude/skills/go-testing-standards/SKILL.md`.
- **`ScorerAlgorithm` AST-detection gap** is a false-positive style issue: the type IS referenced as a return-type from `Scorer.Algorithms()` in many tests, but the AST walker only matches identifier nodes in test files (not return-type matches via interface satisfaction). Closure path: extend the AST walker in `scripts/internal/astwalk/` to recognise return-type usage, OR add an explicit `_ = (fuzzymatch.ScorerAlgorithm)(nil)` reference in a test file.

**Scheduled:** A dedicated v1.x plan **"Phonetic encoder branch-coverage closure"** addressing all three gaps:
1. DoubleMetaphone rule-table branch coverage via SWI-Prolog reference-vector expansion.
2. NYSIIS step-by-step Taft 1970 worked-example coverage.
3. DefaultScorer + NewSWGParams happy-path / error-path explicit unit tests.
4. AST walker `ScorerAlgorithm`-style false-positive fix.
