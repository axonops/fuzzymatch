# Phase 08.5 Plan 10: Q10 — Cross-validation Corpora (character, q-gram, Monge-Elkan) Summary

Three new committed cross-validation corpora (character-tier, q-gram-tier, Monge-Elkan) using jellyfish==1.2.1 and py_stringmatching==0.4.7 as reference oracles — closes Phase 8.5 success criterion 5 and the three Critical findings (test-writer ×3, algorithm-correctness cross-cutting ×3).

## Per-corpus Pin and Case Count

| Corpus | Reference library + pin | Cases | Per-entry surfaces |
|--------|------------------------|-------|---------------------|
| `character` | `jellyfish==1.2.1` | 33 | LevenshteinDistance (int eq), DamerauLevenshteinFullDistance (int eq), JaroScore (1e-9), JaroWinklerScore (1e-9) |
| `qgram` | `py_stringmatching==0.4.7` | 32 | QGramJaccardScore (1e-9), SorensenDiceScore (1e-9), CosineScore (1e-12), TverskyScore (1e-9) — at q=2 mandatory + q=3 optional |
| `monge-elkan` | `py_stringmatching==0.4.7` | 32 | MongeElkanScore (asym) JW (1e-6) + Lev (1e-9); MongeElkanScoreSymmetric JW (1e-6) + Lev (1e-9) |

**Total new test sub-cases produced:** 33 × 4 = 132 (character) + 32 × 4 × 2 (q=2 + q=3 where applicable) ≈ 240 (q-gram) + 32 × 4 = 128 (monge-elkan) = **~500 new sub-test cases**.

## Tasks

### Task 1 — Character-tier (commit `7df48aa`)

Added jellyfish==1.2.1 cross-validation for Levenshtein, full Damerau-Levenshtein, Jaro, JaroWinkler over 33 cases (≥ 30 required). The plan originally specified `DamerauLevenshteinOSADistance` as the surface to cross-validate, but a direct probe revealed jellyfish 1.2.1 ships the **UNRESTRICTED** (Lowrance-Wagner 1975, "true") Damerau-Levenshtein variant — NOT the OSA variant. The discriminating vector `damerau_levenshtein_distance("ca", "abc")` returns 2 in jellyfish (transposition + insert) but 3 in OSA (Boytsov 2011 §3.1). The corpus was therefore re-pointed to `DamerauLevenshteinFullDistance`; the OSA surface remains covered by primary-source reference vectors in `damerau_osa_test.go`. The empty-string Jaro divergence (jellyfish returns 0.0, fuzzymatch returns 1.0 by catalogue convention) was handled via the `variant_divergence` flag pattern inherited from `gen-phonetic-cross-validation.py`.

### Task 2 — Q-gram tier (commit `6924b1b`)

Added py_stringmatching==0.4.7 cross-validation for Q-Gram Jaccard, Sørensen-Dice, Cosine, Tversky over 32 cases (≥ 30 required) at q=2 (mandatory) and q=3 (optional for inputs ≥ 4 chars). The principal divergence handled here was **set vs. multiset semantics**: py_stringmatching's Jaccard/Dice/Cosine/TverskyIndex treat token lists as **sets** (duplicates collapse), whereas fuzzymatch's q-gram surfaces are **multiset** (Ukkonen 1992, min/max over counts). Resolution: restrict corpus inputs to q-gram-unique extractions so set and multiset semantics coincide; the generator's `_assert_qgram_unique` gate enforces this. Multiset semantics are exercised by primary-source reference vectors in `qgram_jaccard_test.go`.

### Task 3 — Monge-Elkan (commit `9c38802`)

Added py_stringmatching==0.4.7 cross-validation for Monge-Elkan with JaroWinkler-inner and Levenshtein-inner over 32 cases (≥ 30 required), exercising both the asymmetric direct surface (`MongeElkanScore`) and the symmetric variant (`MongeElkanScoreSymmetric` — computed in Python as `(forward + reverse) / 2` per the Q3-LOCKED semantics in `06-CONTEXT.md §4`, since py_stringmatching does not expose a symmetric primitive). The principal precision quirk: py_stringmatching's JaroWinkler uses **float32** internally — confirmed by direct probe (`JaroWinkler().get_sim_score("john","johnathan")` returns `0.8888888955116272` vs fuzzymatch's fp64 `0.8888888888888889`, a delta of ~6.6e-9 per token comparison). Monge-Elkan accumulates this drift across the max-of-sim reduction, so the JW-inner surfaces use a **relaxed 1e-6 tolerance**; Levenshtein-inner stays at 1e-9 (integer + division is fp64-clean across both libraries). The plan's `CONTRIBUTING.md` regen-discipline section was added here, listing all 7 committed corpora with their pins / generators / Go tests + a 6-step pin-bump protocol.

**2. [Rule 1 — Bug] jellyfish.jaro_similarity("", "") divergence not in plan**
- **Found during:** Task 1
- **Issue:** Test failure surfaced jellyfish.jaro_similarity returning 0.0 for both-empty input, but fuzzymatch.JaroScore returns 1.0 by documented catalogue convention.
- **Fix:** Adopted the `variant_divergence` pattern from `gen-phonetic-cross-validation.py`: the affected entry stores the fuzzymatch-expected value as the primary score and emits `divergent_jellyfish_jaro` + `divergent_jellyfish_jw` alongside for reviewer triage. The Go-side test asserts the fuzzymatch-expected value (which the corpus stores as the primary value).
- **Commit:** `7df48aa`

**3. [Rule 2 — Auto-add missing critical functionality] py_stringmatching set-vs-multiset divergence not in plan**
- **Found during:** Task 2 (during initial probe of py_stringmatching semantics)
- **Issue:** py_stringmatching's Jaccard/Dice/Cosine/TverskyIndex treat input lists as sets (deduplicating), whereas fuzzymatch's q-gram surfaces are multiset (Ukkonen 1992). Naïve cross-validation would fail on any input with duplicate q-grams (e.g. "AAAA" → ['AA','AA','AA']: py returns set-of-one, fuzzymatch returns multiset-of-three).
- **Fix:** Restricted corpus to q-gram-UNIQUE inputs via the `_assert_qgram_unique` gate in the generator. Documented the divergence in the generator docstring, the Makefile target preamble, the Go test file header, and the CONTRIBUTING.md "Cross-validation Corpora" section. Multiset semantics remain exercised by primary-source vectors in `qgram_jaccard_test.go`.
- **Commit:** `6924b1b`

**4. [Rule 2 — Auto-add missing critical functionality] py_stringmatching float32 JaroWinkler not in plan**
- **Found during:** Task 3 (during initial probe of Monge-Elkan-with-JW-inner outputs)
- **Issue:** py_stringmatching's JaroWinkler uses float32 internally; cross-validating Monge-Elkan with JW-inner accumulated ~6.6e-9 per token-comparison drift across the max-of-sim reduction, exceeding the plan's standard 1e-9 tolerance.
- **Fix:** Relaxed the Monge-Elkan JW-inner tolerance to 1e-6 (plan suggested this might be needed for Cosine; instead it was needed for ME-JW). Documented the float32 precision finding in the generator docstring, the Go test file header, and the Makefile target preamble. Levenshtein-inner remains at 1e-9 (integer + division is fp64-clean).
- **Commit:** `9c38802`

## Decisions Made

1. **jellyfish version-probe via `pip show`** (Task 1): jellyfish 1.2.1 dropped the module-level `__version__` attribute (confirmed: direct probe raises `AttributeError`). The generator falls back to `subprocess.check_output([sys.executable, "-m", "pip", "show", "jellyfish"])` and parses the `Version:` line — mirroring the Metaphone-version probe pattern in `gen-phonetic-cross-validation.py`. The cleaner direct attribute probe (`assert sm.__version__ == PYSM_VERSION`) is used for py_stringmatching, which exposes `__version__` at the module level.

2. **Corpus field rename `damerau_levenshtein_osa_distance` → `damerau_levenshtein_distance`** (Task 1): driven by the jellyfish-mis-labelling finding above. The OSA-style name was misleading because jellyfish actually ships unrestricted Damerau. The new name matches jellyfish's actual surface and the fuzzymatch surface (`DamerauLevenshteinFullDistance`) that the corpus cross-validates.

3. **Tversky parameters fixed at (α=0.5, β=0.5)** (Task 2): matches both Tversky's "symmetric" canonical default and the dispatch slot `AlgoTversky`'s default. The metadata block pins these as load-bearing constants; a future bump would surface via the metadata gate, not via per-entry deltas.

4. **q=2 mandatory + q=3 optional** (Task 2): every entry carries q=2 scores; entries marked `skip_q3: true` (because either input is shorter than q=3, making the q-gram extraction empty) emit `q3: null`. The Go-side test sub-test loop is conditional: q=3 sub-tests are only run when the corpus emits non-null scores.

5. **Cosine relaxed to 1e-12 vs Jaccard/Dice/Tversky at 1e-9** (Task 2): defence-in-depth; the q-gram-unique corpus restriction means count vectors are all-ones in practice, so the reductions produce bit-identical output. The relaxed tolerance guards future numeric paths (e.g. if py_stringmatching introduces a fused-multiply-add anywhere).

6. **Python-side symmetric mean** (Task 3): py_stringmatching does not expose a symmetric Monge-Elkan primitive; the generator computes `(forward + reverse) / 2` in Python and stores both forward (`monge_elkan_asymmetric_*`) and the derived symmetric mean (`monge_elkan_symmetric_*`). The Go side cross-validates against both — the asymmetric `MongeElkanScore` and the symmetric `MongeElkanScoreSymmetric`.

7. **Monge-Elkan inner coverage limited to {AlgoJaroWinkler, AlgoLevenshtein}** (Task 3): the default inner (load-bearing) plus a representative of the integer-distance / max-len family. The other 12 permitted inners (DL-OSA, DL-Full, Hamming, Jaro, Strcmp95, SWG, LCSStr, Q-Gram Jaccard, Sørensen-Dice, Cosine, Tversky, Ratcliff-Obershelp) are exercised by primary-source reference vectors in `monge_elkan_test.go` — avoiding the fp32-vs-fp64 precision investigation across 14 inner surfaces.

8. **CONTRIBUTING.md "Cross-validation Corpora" section as a single 7-corpus table** (Task 3): listing the four pre-existing corpora (swg, ratcliff-obershelp, token-ratios, phonetic) alongside the three new ones (character, qgram, monge-elkan) keeps the regen discipline in one place. The 6-step pin-bump protocol is shared across all 7 corpora.
