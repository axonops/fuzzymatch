## 2. Findings

### 2.1 Distributed source (root module)

| Surface | Result |
|---|---|
| `.go` files under root (excluding `tests/`) | 190 files, **0 matches** for `HashiCorp` / `hashicorp` / `MPL` |
| Root `go.mod` direct requires | `golang.org/x/text v0.37.0` only (BSD-3-Clause; on the allowlist per CLAUDE.md and `make verify-deps-allowlist`) |
| Root `go.mod` indirect requires | None (only `golang.org/x/text` and its zero-transitive surface) |
| Imports of `github.com/hashicorp/*` in distributed source | **0** |

**Conclusion (distributed):** the distributed `fuzzymatch` Go module is Apache-2.0 throughout. No MPL-2.0-derived utility code present. No HashiCorp imports. The `LICENSE` file (Apache-2.0) is correct.

### 2.2 Test-only BDD module (`tests/bdd/go.mod`)

The BDD module pulls in three HashiCorp packages **indirectly** via godog's internal storage layer:

| Indirect Dep | Version | Licence | Pulled in by |
|---|---|---|---|
| `github.com/hashicorp/go-immutable-radix` | v1.3.1 | MPL-2.0 | `cucumber/godog/internal/storage` → `hashicorp/go-memdb` → here |
| `github.com/hashicorp/go-memdb` | v1.3.4 | MPL-2.0 | `cucumber/godog/internal/storage` |
| `github.com/hashicorp/go-uuid` | v1.0.2 | MPL-2.0 | transitive of `go-memdb` |
| `github.com/hashicorp/golang-lru` | v0.5.4 | MPL-2.0 | transitive of `go-memdb` (via go-immutable-radix) |

**Conclusion (BDD test-only):** the four HashiCorp dependencies are pulled in transitively by `godog` for its internal scenario-storage layer. They are present only in `tests/bdd/go.mod` — **the BDD module is a separate Go module that is never distributed**, never referenced from the root `go.mod`, never imported by any distributed `.go` file. The structural isolation matches CLAUDE.md's "Test dependencies (BDD only): godog, goleak, testify — all in `tests/bdd/go.mod`, structurally isolated" rule.

MPL-2.0 in test-only dependencies is acceptable: the MPL-2.0 file-level copyleft applies only to modifications of MPL-2.0 source files, not to derivative works that link against an MPL-2.0 dependency. Since fuzzymatch does not modify the HashiCorp source files and does not distribute the BDD module to consumers, no MPL-2.0 obligation propagates to the distributed Apache-2.0 surface.

### 2.3 No MPL-2.0 utility shims in distributed code

A common anti-pattern in Go libraries is to embed small utility helpers ("shims") copied from HashiCorp's utility crates (`go-immutable-radix`, `golang-lru`, `go-memdb`) into the distributing project's own source — usually for performance or to avoid the dependency. Such embedded code would inherit MPL-2.0 file-level obligations and would violate the Apache-2.0 distribution contract.

**Audit result:** no such shims exist. The distributed source contains zero HashiCorp-derived code. Every utility (LRU caches, immutable maps, UUIDs, memdb-style queries) is either implemented from scratch or sourced from the Go standard library / `golang.org/x/text`.
