## Shape-extension for code-returning algorithms

The plan flagged that DoubleMetaphone (returns two code strings), NYSIIS / Soundex (return one code string), and MRA (returns a binary match decision) might require framework extension. **No extension was needed:**

- `DoubleMetaphoneScore(a, b string) float64` — wraps the two-code comparison into the canonical 0.0 / 1.0 binary (per `dispatch_double_metaphone.go`).
- `SoundexScore(a, b string) float64` — wraps the four-character-code comparison.
- `NYSIISScore(a, b string) float64` — wraps the code comparison.
- `MRAScore(a, b string) float64` — wraps `MRACompare`'s threshold rule into the 0.0 / 1.0 binary.

The standard `goldenAlgorithmEntry{Name, Algorithm, A, B, ExpectedScore float64}` shape applies unchanged to all four.

## Decisions Made

1. **Drop `derivation_note` and `deviation_note` informational fields.** These fields existed in the hand-committed JSONs (`partial_ratio.json`, `token_set_ratio.json`) but `goldenAlgorithmEntry` has no place for them and the canonical marshal helper does not preserve unknown fields. The rationale they captured (RapidFuzz issue #110 deviation; 06-RESEARCH.md Pitfall 3 KEYSTONE three-region regression gate) is documented inline in the test godocs and in the linked CONTEXT.md / RESEARCH.md sections — the informational notes were redundant once the producing tests exist.
2. **Sort all 23 staging files alphabetically by Name.** The 9 hand-committed JSONs were in source-order (e.g., DoubleMetaphone's `both_empty` first, `Catherine_Katherine_exact` later); the 14 pre-existing staging files all use `sort.Slice` by Name. Standardising on alphabetical-by-Name across all 23 keeps the merge step uniform (no per-algorithm sort-key special case) and matches the template precedent.
3. **MongeElkan staging entries score via `MongeElkanScoreSymmetric`** with the LOCKED default inner `AlgoJaroWinkler` and `DefaultNormalisationOptions()` — the variant bound to `dispatch[AlgoMongeElkan]` per `dispatch_monge_elkan.go`. The asymmetric direct surface (`MongeElkanScore`) is not exercised at the staging-file level (it is exercised by per-direction asymmetry unit tests in `monge_elkan_test.go`).
